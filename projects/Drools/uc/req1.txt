<p>This has 3 problems:</p>

<ul class= alternate  type= square >
	<li><b>Loss of precision</b>: the long sum `1881617265586265321L` will incorrectly return `1.88161726558626534E18`, so `13` too much! The BigDecimal sum of `0.09` and `0.01` will also be incorrect.</li>
	<li><b>Loss of performance</b>: Summing with a Double total is significantly slower than summing with a Long total or an Integer total.</li>
	<li>Example complexity (minor, not all of us agree on this argument): the use of `Number` is an abstraction that doesn t bring any value to the use case. It s worthless complexity.</li>
</ul>


<p>Solution proposal A) for 7.0 as discussed with Mario:<br/>
Based on the argument type to the sum function, the compiler selects a different sum function implementation. This is similar to java overloading mechanism, where `System.out.println(1)` selects a different method implementation than `System.out.println(2.0)`.</p>
